## 二、优化决策

### 2.1 Mock 系统决策

#### 决策选项评估

| 选项 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| A. 启用 Mock | 前端独立开发测试 | 需修改request.ts，增加复杂度 | ⭐⭐ |
| B. 删除 Mock | 代码简洁，减少维护 | 失去Mock能力，依赖后端 | ⭐⭐⭐ |
| C. 保留待启用 | 保持灵活性，代码已存在 | 占用空间，配置混乱 | ⭐⭐⭐⭐⭐ |

#### ✅ 最终决策：选择 C - 保留但标记为"待启用"

**理由**:
1. Mock 系统已完整实现，删除浪费
2. 后端 API 尚未完全就绪，保留备用
3. 仅需修改配置和添加注释，风险最低
4. 为后续前端独立开发提供便利

---

### 2.2 状态管理决策

#### 决策选项评估

| 选项 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 集中迁移到 Store | 状态共享，数据一致 | 工作量大，需重构28个页面 | ⭐⭐ |
| 保留本地状态 | 无需修改，风险低 | 状态分散，可能不一致 | ⭐⭐⭐⭐ |
| 混合模式 | 按需迁移，兼顾灵活性 | 需明确迁移规则 | ⭐⭐⭐⭐⭐ |

#### ✅ 最终决策：选择混合模式 - 优先迁移高频共享状态

**迁移策略**:
1. **必须迁移**: 用户信息、学生列表、订单列表
2. **建议迁移**: 消息列表、考勤记录
3. **保留本地**: 表单临时状态、UI交互状态
