## 1. 项目背景与建设目标

### 1.1 项目背景

在“双减”与“5+2”政策持续推进背景下，学校课后服务从临时性工作逐渐转向常态化、规范化和数字化管理。现实场景中，家长关注“孩子是否安全到达、是否完成作业、何时接回”；学校关注“课后服务是否可监管、是否可量化”；机构关注“服务运营是否高效、是否便于排课和结算”。

传统线下管理模式普遍存在以下问题：
- 信息依赖微信群、纸质登记和电话沟通，效率低且难以追踪；
- 家长、学校、托管机构之间信息割裂，业务闭环不完整；
- 高峰期签到、异常反馈和消息通知易发生拥堵；
- 缺乏面向未成年人保护的数据安全与权限隔离机制；
- 无法形成可持续沉淀的经营、监管与成长数据资产。

### 1.2 建设目标

本项目拟建设一款基于 HarmonyOS 原生应用形态的学生智慧托管系统，以“学校、家长、机构三方协同”为核心，围绕学生课后托管全生命周期，构建集预约、审核、排课、签到、辅导、反馈、统计、安全于一体的智慧平台。

### 1.3 量化目标

- 用户满意度：**≥ 85%**
- 核心接口响应时间：**≤ 2 秒**
- 页面加载时间：**≤ 3 秒**
- 支撑放学高峰期“集中签到并发”场景
- 实现未成年人数据的分级分类安全防护

## 2. 产品顶层蓝图

### 2.1 总体业务闭环

系统整体运作闭环如下：

1. **家长预约**：家长通过 HarmonyOS 客户端或元服务卡片查看托管服务、提交预约申请并完成支付。  
2. **机构接管**：机构/学校审核订单，确认服务资格、班次资源和名额。  
3. **智能排课**：系统按照年级、班次、教师、容量和时间规则生成课程班次。  
4. **签到执行**：学生到达后完成签到，系统同步向家长反馈状态。  
5. **作业辅导**：教师根据任务清单组织课后作业辅导与看护。  
6. **家校反馈**：教师提交作业反馈，家长通过 IM、时间线和卡片查看并确认。  
7. **签退接回**：家长或授权接送人完成签退接回，形成闭环记录。  
8. **数据分析**：沉淀考勤、订单、财务、反馈、绩效等数据，服务监管与运营决策。

### 2.2 产品价值闭环

- 对家长：可视、可感知、可追踪  
- 对学校：可监管、可考核、可分析  
- 对机构：可运营、可排课、可结算  
- 对教师：可执行、可反馈、可减负  
- 对平台：可沉淀、可治理、可扩展

### 2.3 角色分工

| 角色 | 核心职责 |
|---|---|
| 家长 | 预约下单、支付、查看状态、接收反馈、接回学生 |
| 学生 | 参与签到、托管学习、作业辅导、签退 |
| 教师/看护老师 | 签到核验、辅导执行、反馈上报、异常处理 |
| 机构管理员 | 服务维护、订单审核、排课管理、财务管理 |
| 学校管理员 | 监管托管服务质量、统计分析、审批特殊事件 |
| 平台管理员 | 权限配置、安全审计、系统配置、基础治理 |

## 3. 建设原则与设计思路

### 3.1 建设原则

1. **业务闭环优先**：先打通预约、审核、签到、反馈、统计主链路。  
2. **安全合规优先**：未成年人数据保护是系统设计底线。  
3. **高频体验优先**：围绕家长高频入口和放学高峰场景优化。  
4. **端云协同优先**：充分发挥 HarmonyOS 端侧能力与云端治理能力。  
5. **可扩展优先**：支持未来拓展 AI 排课、成长画像、更多机构接入。

### 3.2 设计思路

采用“鸿蒙端原生交互 + 云端服务化治理 + 缓存与消息解耦 + 数据安全分层防护”的总体思路：
- 端侧通过 ArkUI/ArkTS 构建高性能页面；
- 云侧通过 API 网关、业务服务、Redis、MQ、MySQL 形成稳定后端；
- 通过卡片、推送和分布式能力提升体验差异化；
- 通过统一权限、安全审计和敏感数据保护实现合规落地。

## 4. 总体架构设计

### 4.1 架构分层

系统总体可划分为五层：

1. **终端接入层**：家长端、教师端、管理端、元服务卡片。  
2. **网关接入层**：统一鉴权、限流、日志、路由和安全拦截。  
3. **业务服务层**：认证、学生、服务、订单、排课、考勤、作业、消息、报表。  
4. **平台能力层**：Redis、MQ、对象存储、日志中心、审计中心。  
5. **数据存储层**：MySQL 主业务库、统计聚合库、归档库。

### 4.2 服务划分建议

- 认证与权限服务
- 学生档案服务
- 托管服务产品服务
- 订单与支付服务
- 排课服务
- 考勤与预警服务
- 作业与反馈服务
- 消息通知服务
- 报表统计服务
- 安全审计服务

### 4.3 技术栈

- 开发环境：DevEco Studio

- UI 框架：ArkUI
- 接口协议：RESTful API / JSON

## 5. 基于鸿蒙特性的深度设计

### 5.1 元服务（服务卡片）提升家长端体验

#### 5.1.1 今日托管卡片

展示当日学生托管摘要信息：
- 学生姓名、班次名称、托管时段
- 当前状态：待签到 / 已签到 / 辅导中 / 待接回 / 已签退
- 最新动态：签到时间、作业完成率、教师反馈摘要
- 快捷操作：请假、查看接送码、进入消息会话

**设计价值**：家长无需完整打开应用，即可在桌面或负一屏获得最关键的孩子状态信息。

#### 5.1.2 异常提醒卡片

在发生未签到、迟到、错签、未授权接送、作业未确认等事件时，卡片动态更新，展示：
- 异常类型与触发时间
- 风险等级
- 建议处理动作
- 快捷进入教师会话或事件详情

**设计价值**：提高异常处置时效，将风险提醒前置到系统入口。

### 5.2 基于分布式流转/分布式软总线的创新场景

#### 场景一：家长预约跨设备流转

家长在手机端浏览服务并发起预约后，可一键流转到平板设备继续完成支付确认。已选学生、服务套餐、日期区间、优惠信息同步带入，无需重复输入。

**创新价值**：实现碎片化发起 + 大屏深度确认的连续体验。

#### 场景二：教师手机—班级大屏协同签到

教师手机发起签到任务后，通过分布式软总线将签到名单、异常名单和签到进度实时投射到班级大屏或平板。教师移动核验单个学生，大屏同步展示整班状态。

**创新价值**：适配放学高峰期点名场景，提升集中签到效率和现场可视化能力。

### 5.3 鸿蒙能力与业务结合点

- **ArkUI**：高性能构建多端统一体验页面。  
- **ArkTS**：统一业务建模，提升可维护性。  
- **服务卡片**：承载家长高频状态触达。  
- **推送通知**：支撑签到、审核、异常、签退提醒。  
- **分布式能力**：用于跨设备任务流转和协同展示。  
- **系统安全能力**：为隐私数据提供沙箱、密钥、权限和可信能力支撑。

## 6. 核心业务流程时序

### 6.1 学生签到与异常预警触发

#### 6.1.1 时序逻辑

1. 家长完成预约并支付，系统生成订单与待执行班次。  
2. 考勤服务预生成待签到记录。  
3. 学生到达托管点，教师终端/闸机终端发起签到。  
4. 系统校验学生身份、订单状态、班次匹配、时间窗、设备可信性。  
5. 校验通过后写入签到记录，更新状态为“已签到”。  
6. 事件中心异步推送至：
   - 家长消息中心
   - 学生动态时间线
   - 今日托管卡片
7. 如发生超时未签到、非本班次签到、重复签到、非授权签到、围栏异常等，则触发异常预警。  
8. 预警中心分级通知教师、机构管理员与家长。  
9. 人工复核后形成处理闭环并记录结果。

#### 6.1.2 设计要点

- 主链路快速返回，异常与通知异步处理；
- 对同一学生同一班次设置唯一性约束；
- 预警采用分级策略，减少误报和通知打扰。

### 6.2 作业辅导与家校反馈

#### 6.2.1 时序逻辑

1. 系统根据教师录入或家长上传生成作业任务。  
2. 教师查看当日作业清单并组织辅导。  
3. 教师记录完成进度、错题情况、表现标签和附件。  
4. 系统形成作业反馈并推送家长。  
5. 家长查看反馈，可确认、留言或发起补充沟通。  
6. 如出现争议，进入复核/审批流。  
7. 最终结果沉淀到时间线、学习画像和绩效统计。

#### 6.2.2 设计要点

- 反馈支持草稿、已提交、家长已确认、复核完成等状态流转；
- 图片附件需要权限控制与脱敏访问；
- 家长确认需全程留痕。

## 8. 系统级安全防护方案

### 8.1 安全设计原则

- 最小必要采集
- 分级分类保护
- 全生命周期防护
- 端云协同治理
- 可审计、可追踪、可回溯

### 8.2 学生信息安全策略

#### 数据脱敏
- 学生姓名采用局部脱敏展示，如“张*”“李**”；
- 手机号、身份证号、住址等仅授权角色可见；
- 默认导出脱敏报表，明文导出需审批。

#### 传输加密
- 全接口统一使用 HTTPS/TLS1.2+；
- 接送码、监护关系授权信息使用业务层二次加密；
- IM 与推送连接使用令牌鉴权和加密信道。

#### 存储加密
- 敏感字段列级加密；
- 附件、照片、回执材料采用对象存储加密；
- 端侧缓存仅保留必要摘要，并放在应用沙箱中。

### 8.3 支付数据保护策略

- 支付域与业务域逻辑隔离；
- 不存储完整银行卡等强敏感信息；
- 回调验签、防重放、防篡改；
- 高风险退款与补单操作启用强审计和二次确认。

### 8.4 通信数据保护策略

- 会话级鉴权与消息签名；
- 图片、附件使用临时访问凭证；
- 对教师与家长沟通记录实施权限分级访问；
- 敏感词识别与告警。

### 8.5 结合 HarmonyOS 底层能力

- 应用沙箱保护本地数据；
- 密钥管理能力存储本地密钥与令牌；
- 分布式访问控制保护跨设备流转数据；
- 设备可信认证用于签到终端校验；
- 对摄像头、定位、NFC 等权限进行最小化申请与动态提示。

## 9. 高并发与性能优化策略

### 9.1 性能目标

- 核心响应时间 ≤ 2 秒
- 页面加载 ≤ 3 秒
- 高峰期签到链路稳定可用

### 9.2 端侧优化

#### ArkUI 渲染优化
- 列表懒加载与可视区渲染；
- 高频组件复用；
- 细粒度状态更新，减少整页重绘；
- 骨架屏和资源预取优化首屏体验。

#### 页面请求优化
- 首页摘要先行、详情后置；
- 本地缓存 + 增量刷新；
- 列表分页与游标加载；
- 非强一致信息异步刷新。

### 9.3 云侧优化

#### Redis 缓存
- 缓存服务列表、班次摘要、学生当日状态、首页卡片摘要；
- 对签到热点数据设置短 TTL 与事件失效机制。

#### MQ 削峰
- 签到主链路只做关键校验与落库；
- 异常预警、消息推送、时间线写入、统计更新异步化；
- 高峰期对通知任务按优先级队列调度。

#### 数据库优化
- 大表分区/分表；
- 高并发接口幂等控制；
- 索引覆盖学生、班次、日期、状态等常用维度；
- 读写分离与限流降级策略。

### 9.4 高峰期签到架构路径

1. 请求进入 API 网关。  
2. 网关完成鉴权、限流和参数校验。  
3. 考勤服务优先查询 Redis 热数据。  
4. 主库写入签到结果并立即响应。  
5. MQ 异步触发通知、时间线、预警与统计。  
6. 报表服务异步消费事件流完成汇总。

- `guardian_relation`
- `service_product`
- `session_schedule`
- `homework_task`
- `homework_feedback`
- `student_timeline`

- HomeworkTask 与 HomeworkFeedback 为一对多；
- User 与 MessageRecord 通过发送者/接收者形成一对多关系。

## 12. 实施路线与阶段规划

### 第一阶段：基础能力建设
