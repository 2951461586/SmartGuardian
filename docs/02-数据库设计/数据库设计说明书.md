## 1. 设计目标

数据库设计用于支撑用户认证、学生档案、托管服务、订单审核、排课、考勤、作业反馈、消息互动、时间线与统计分析等核心场景，并与现有 SQL 建表脚本保持一致。

## 2. 数据库选型

- 数据库：MySQL 8.0+
- 字符集：utf8mb4
- 排序规则：utf8mb4_unicode_ci
- 存储引擎：InnoDB

## 3. 设计原则

1. 满足业务实体完整建模。  
2. 兼顾数据一致性与高频查询性能。  
3. 对敏感字段预留脱敏、加密与审计能力。  
4. 统一逻辑删除和审计字段。  
5. 支撑高峰签到与报表统计扩展需求。

## 4. 核心表说明

### 4.1 用户与学生域
- `user`：存储家长、教师、管理员账号信息。
- `student`：存储学生基本档案信息。
- `guardian_relation`：存储学生与监护人绑定关系。

### 4.2 托管服务域
- `service_product`：托管服务产品定义。
- `order_info`：家长预约下单与审核数据。
- `session_schedule`：班次、排课与教师分配数据。

### 4.3 执行与互动域
- `attendance_record`：签到签退与异常记录。
- `homework_task`：作业任务数据。
- `homework_feedback`：辅导反馈与家长确认数据。
- `message_record`：消息会话记录。
- `student_timeline`：学生动态时间线。

### 4.4 告警与退款域
- `alert_record`：系统告警与异常预警记录。支持考勤异常、安全隐患、学业表现、行为问题、费用到期、系统提醒等类型。状态流转包括：ACTIVE（活动）→ ACKNOWLEDGED（已确认）→ RESOLVED（已解决）/ DISMISSED（已忽略）。
- `refund_record`：退款申请与处理记录。支持退款申请、审核、处理、完成全流程。退款原因类型包括：学生转学、服务质量、时间变动、个人原因、其他。

### 4.5 支付与财务域
- `payment_record`：支付订单与回调记录。
- `finance_report`：财务收支数据。

## 5. 关键关系

- 用户与学生通过 `guardian_relation` 建立多对多关系。  
- 学生与订单、考勤、作业、时间线均为一对多关系。  
- 服务产品与订单、班次为一对多关系。  
- 作业任务与作业反馈为一对多关系。  
- 消息记录通过发送人与接收人与用户关联。
- 学生与告警记录为一对多关系（一个学生可能有多条告警）。
- 订单与退款记录为一对多关系（一个订单可能有多笔退款记录）。
- 告警记录与用户（确认人、解决人）为多对一关系。

## 6. 索引与约束设计

增强版 SQL 中已包含：
- 用户账号、用户名、手机号唯一约束
- 学生编号唯一约束
- 订单号唯一约束
- 学生+日期+班次唯一签到约束
- 基于角色、状态、日期、班次、会话的复合索引

## 7. 审计与逻辑删除

核心表普遍包含：
- `is_deleted`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`

该设计便于保留历史数据、支持追溯与审计。

## 8. 性能建议

- 对考勤、消息、时间线等大表进行分区/归档。  
- 对高频查询条件建立覆盖索引。  
- 结合 Redis 缓存热点业务数据。  
- 统计任务异步化，避免阻塞主业务链路。

## 9. 安全建议

- 对手机号、证件号、健康信息等敏感字段采用列级加密。  
- 数据导出默认脱敏处理。  
- 数据库账户按职责进行最小权限隔离。  
- 审计日志与业务库分域管理。
