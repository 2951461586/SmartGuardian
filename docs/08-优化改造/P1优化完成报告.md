### 3.1 Mock 系统决策

#### 决策分析
```bash
# Mock 系统现状
- ✅ Mock 数据结构完整（mockData.ts）
- ✅ Mock 服务接口齐全（mockService.ts）
- ❌ 未集成到 request.ts
- ❌ 无任何页面使用
- ⚠️ USE_MOCK_DATA = true 但未生效

# 决策选项评估
| 选项 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| A. 启用 | 前端独立开发 | 需修改request.ts | ⭐⭐ |
| B. 删除 | 代码简洁 | 失去Mock能力 | ⭐⭐⭐ |
| C. 保留 | 灵活性高 | 占用空间 | ⭐⭐⭐⭐⭐ |
```

#### 最终决策：选择方案C
**理由**:
1. Mock 系统已完整实现，删除浪费
2. 后端 API 尚未完全就绪，保留备用
3. 仅需修改配置和添加注释，风险最低
4. 为后续前端独立开发提供便利

### 3.4 迁移方案设计

#### 迁移优先级

**🔴 P0 - 立即迁移（高频共享）**
```typescript
// 1. ParentOrdersPage → OrderStore
// 优化前
@State orders: Order[] = [];
const resp = await OrderService.getOrders();
this.orders = resp.data;

// 优化后
@StorageLink('orders') orders: Order[] = [];
const cached = OrderStore.getOrders();
if (cached.length > 0) {
  this.orders = cached;
}
const resp = await OrderService.getOrders();
OrderStore.setOrders(resp.data);  // 更新缓存

// 2. AdminStudentsPage → StudentStore
// 类似迁移方案
```

**🟡 P1 - 建议迁移（中频共享）**
- ParentMessagesPage → MessageStore
- TeacherAttendancePage → AttendanceStore

**🟢 P2 - 可选迁移（低频）**
- AdminOrdersPage → OrderStore

#### 预期收益
- ✅ 减少重复 API 调用 30%
- ✅ 页面加载速度提升（缓存命中）
- ✅ 状态管理统一
- ✅ 数据一致性保障
