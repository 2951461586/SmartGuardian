# SmartGuardian Cloud DB 数据模型迁移方案

> 日期：2026-04-25  
> 目标：把现有 `MySQL` 主数据、交易数据、事件数据重排到 `Cloud DB` 模型

> 退役状态：截至 2026-04-26，`backend/` 与 MySQL 本地运行链路已退役；本文中的 MySQL 表仅作为历史迁移映射和 `legacyId` 对账来源。

## 已落地原则

✔ 以“业务集合”替代“关系表 join 为中心”的后端建模。  
✔ 以“读模型前置”替代“前端多接口拼装”。  
✔ 以“租户/机构/学校上下文”作为每条业务数据的基础字段。  
✔ 保留 `legacyId` 兼容历史 MySQL 数据迁移和对账。

## 统一字段规范

✔ 每个集合统一包含：`docId`、`legacyId`、`tenantId`、`orgId`、`schoolId`、`createdAt`、`updatedAt`。  
✔ 交易集合统一包含：`status`、`operatorId`、`operatorName`、`sourceChannel`。  
✔ 聚合集合统一包含：`payloadVersion`、`generatedAt`、`expiresAt`。  
✔ 删除统一走软删：`isDeleted` + `deletedAt`，不直接物理删主业务数据。  

## 迁移策略

### 1. 主数据先迁

✔ 先迁 `user / org / student / guardian / service_product`。  
✔ 保持 `legacyId -> docId` 映射表，保证旧订单、旧消息还能回查。  

### 2. 交易数据再迁

✔ 再迁 `order / session / attendance / homework / refund`。  
✔ 交易集合保留冗余展示字段：`studentName`、`teacherName`、`serviceProductName`，减少 Cloud DB 读时拼装。  

### 3. 事件与读模型最后迁

✔ `message / alert / timeline / card / workbench / report` 统一作为读模型或事件模型。  
✔ 报表和工作台快照由云函数异步生成，不让前端临时拼表。  

## 安全与权限

✔ 家长仅可读自己的学生、订单、消息、反馈。  
✔ 教师仅可读自己班次相关学生、考勤、作业。  
✔ 机构管理员可读本机构全量运营数据。  
✔ 平台管理员单独保留跨机构运营视角，不直接给普通云函数域默认开启。  

## 与现有前端模型的对应

✔ `UserInfo` 对应 `sg_user_profile`。  
✔ `Student` 对应 `sg_student_profile`，并保留 `guardianName`、`className` 读时冗余。  
✔ `Order` / `SessionSchedule` / `AttendanceRecord` / `HomeworkTask` / `MessageRecord` 等现有 ArkTS 模型可继续沿用，先不要求前端全面改 DTO。
