# 2026-04-24 全栈新增重构总控

## 1. 本轮定位

本轮按“先建总控，再切模块”的方式推进 SmartGuardian 全栈新增与重构。项目现状已经具备 HarmonyOS ArkTS 三端页面、Mock 数据层、Spring Boot + MySQL 后端、OpenAPI 文档和登录页视觉方向，因此不做一次性推倒重写，而是以顶层设计为主线，把现有能力整理为可迭代、可验证的全栈工程。

## 2. 设计锚点

- 视觉基准：沿用当前登录页的浅灰蓝页面底、白色信息卡片、柔和边框、轻阴影和三端角色色。
- 家长端：安全蓝，聚焦今日托管、预约支付、接送码、作业反馈和成长时间线。
- 教师端：成长绿，聚焦高峰签到、班次名单、作业辅导、异常处置和家校消息。
- 机构端：托管橙，聚焦订单审核、服务产品、班次排课、学生档案、告警、退款和经营报表。
- 业务主链路：预约下单 -> 审核排班 -> 签到执行 -> 作业反馈 -> 接回签退 -> 统计沉淀。
- 安全底线：未成年人数据脱敏、最小权限、JWT 鉴权、审计留痕、导出审批。

## 3. 第一轮已落地切片

新增统一工作台能力清单接口：

- 后端：`GET /api/v1/workbench/manifest`
- 前端：`WorkbenchService.getManifest()`
- Mock：补齐同路径响应，保持 Mock 与真实后端同构
- 用途：让三端首页、底部导航、模块入口、验收清单和后端能力版本有统一来源

这个接口不是最终业务页面，但它是后续“整端重构”的锚点：每个角色能看到哪些模块、当前采用什么视觉语义、主流程是什么，都可以从同一份 manifest 读取和校验。

## 4. 后续分阶段任务

### R1：全栈骨架收口

- 前端把 `MainPage` 的导航、首页快捷入口和空状态接入 `WorkbenchManifest`
- 后端为 manifest 增加数据库/配置化来源，支持机构差异化开关
- OpenAPI 补齐 workbench 分组
- 验收：三端登录后工作台模块清单与后端返回一致

### R2：家长端闭环

- 服务浏览、下单、支付、订单详情、退款申请走真实接口
- 今日托管卡片接入签到、作业、消息、告警聚合数据
- 成长时间线合并考勤、作业反馈、老师评价
- 验收：家长可完整走通预约到接回的核心闭环

### R3：教师端闭环

- 今日班次、学生名单、扫码签到、签退、异常处理真实联调
- 作业任务、反馈、家长确认状态统一
- 教师消息中心围绕学生和班次聚合
- 验收：放学高峰签到主链路响应稳定，异常可闭环处理

### R4：机构端闭环

- 服务产品、订单审核、排课、学生档案管理打通
- 告警、退款、报表进入可运营状态
- 数据导出增加权限与审计
- 验收：机构端可完成服务运营与财务/安全治理

### R5：工程化与发布

- OpenAPI 与 ArkTS 模型校验
- 后端测试覆盖核心 service 和 controller
- Mock 数据降级为开发/演示能力，真实后端为默认验收链路
- 部署、巡检、上线检查清单更新
