# SmartGuardian 前端 UI 深度重构落地方案

> 日期：2026-04-26
> 目标：基于 HarmonyOS UI 设计与 ArkUI 组件化原则，完成智慧学生托管系统三端前端界面的系统化重构。

## 背景判断

- 当前工程已经具备 `HarmonyDesign.ets`、`RoleTheme.ets`、`components/common/*` 等基础雏形。
- 但页面层仍存在大量硬编码颜色、阴影、间距和一次性布局，组件体系尚未形成真正约束。
- 高优先级页面仍然偏大：报表、首页、表单、排课等主链路页面承担了过多视觉与状态逻辑。
- 空态、错误态、加载态、重试态已有组件，但尚未形成统一交互协议。

## 本轮总目标

1. 收口设计令牌层，建立统一视觉入口。
2. 补齐统一页面骨架，替代页面内重复布局代码。
3. 按场景沉淀模板，而不是继续按页面逐个修补。
4. 完成首页、报表、表单页第一批接入。
5. 建立 UI 治理门禁，避免后续反弹。

## 分阶段落地

### P0 设计底座

- 统一颜色、阴影、边框、圆角、间距、内容宽度、响应式断点。
- 强化深色主题资源，避免页面直接依赖亮色常量。
- 新增响应式布局与双栏切换基础组件。
- 新增统一页面骨架组件，统一头部、内容区、底部区域、最大宽度与滚动行为。

### P1 场景模板

- 仪表盘模板：家长首页、教师首页、机构首页。
- 数据分析模板：报表、概览、趋势与筛选类页面。
- 表单流程模板：登录、注册、请假、退款、下单等录入型页面。
- 统一 loading / empty / error / retry 容器，形成页面状态协议。

### P2 首批接入页面

- 登录页、注册页。
- 家长首页、教师首页、机构首页。
- 机构报表页。

### P3 治理与扩散

- 引入 UI 治理脚本，扫描页面层硬编码视觉值。
- 与页面体量预算联动纳入 `run-refactor-gates.ps1`。
- 后续扩展到订单、消息、详情、资料页。

## 执行清单

1. 先收设计令牌层。
2. 补一套统一页面骨架。
3. 按场景沉淀三类页面模板。
4. 首页做二次拆分，不再让首页承担全部视觉表达。
5. 报表页单独做“数据可视化规范”。
6. 表单页统一成“字段系统”。
7. 做响应式和大屏适配，不再默认手机单栏。
8. 补暗色主题和可访问性基线。
9. 把 loading / empty / error / retry 做成全局交互协议。
10. 建立 UI 治理门禁。

## 代码落点

- 设计令牌：`entry/src/main/ets/common/`
- 页面骨架：`entry/src/main/ets/components/common/PageScaffold.ets`
- 状态协议：`entry/src/main/ets/components/common/StatefulContainer.ets`
- 表单系统：`entry/src/main/ets/components/common/FormFields.ets`
- 首批接入页面：`pages/LoginPage.ets`、`pages/RegisterPage.ets`、`pages/*HomePage.ets`、`pages/admin/AdminReportsPage.ets`
- UI 门禁：`scripts/check-ui-governance.ps1`

## 资源策略

- 优先复用现有 SVG 图标体系。
- 对于空态、引导态和装饰性插图，补充项目内位图资源，减少默认图标堆叠和 AI 味儿重的通用画面。
- 新增位图资源必须进入工程资源目录，不允许长期依赖外部临时路径。

## 验收标准

- 页面层新增代码不再直接写十六进制颜色作为常规视觉值。
- 首页、报表、表单页接入统一骨架与状态协议。
- 主页面在深色主题下不出现大面积亮色背景断裂。
- 平板或大宽度场景下，关键页面具备内容最大宽度约束或分栏布局。
- `run-refactor-gates.ps1` 新增 UI 治理检查并通过。
