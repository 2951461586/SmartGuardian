# SmartGuardian Mock 验证后续优化执行清单

> 更新日期：2026-04-21  
> 适用阶段：`DEV_MOCK` 基线验证完成后，进入真实联调与质量收口阶段

---

## 1. 当前结论

- 当前项目默认环境仍为 `DEV_MOCK`，配置位于 `entry/src/main/ets/config/api.config.ts`。
- `entry/src/main/ets/services/mock/mockService.ts` 已覆盖家长端、教师端、管理端主链路所依赖的主要接口模块，可支撑页面演示、流程走查与课堂答辩。
- 目前“接口可用”结论仅适用于 Mock 路径，不等同于真实后端已打通。
- 下一阶段应从“Mock 可跑”转向“真实后端可联、文档一致、页面稳定、测试可回归”。

---

## 2. 优先级划分

### P0：立即执行

1. 切换真实联调环境
   - 配置 `ApiConfig.CURRENT_ENV = TEST_REAL`
   - 补齐 `TEST_BASE_URL`
   - 准备家长/教师/管理员测试账号
   - 验证登录、鉴权、401 回跳

2. 跑三端主链路真实联调
   - 家长端：登录 → 首页 → 服务详情 → 下单 → 订单详情 → 支付/退款 → 消息/作业
   - 教师端：登录 → 今日班次 → 考勤管理 → 扫码签到 → 作业反馈 → 消息
   - 管理端：登录 → 首页 → 学生管理 → 订单审核 → 报表查看 → 消息

3. 收口接口文档与 OpenAPI
   - 统一 `Cards` 路径命名
   - 统一 `Sessions Today` 语义
   - 补齐新增接口到 OpenAPI 与清单文档

### P1：高优先级

4. 统一服务层出口
   - `attendance.ts` 补齐请假列表、取消请假、统计查询
   - 评估并收敛 V1/V2 双份服务实现，避免并行维护
   - 统一分页与返回模型定义

5. 建立最小 API 回归集
   - 至少覆盖 Auth、Students、Orders、Attendance、Messages、Refunds、Reports
   - 优先做可重复执行的 smoke case
   - 将 Mock 验证与真实联调验证拆成两个固定清单

6. 页面级错误反馈补齐
   - 当前多处仍仅 `console.error` 或静默失败
   - 对加载失败、提交失败、跳转失败补齐用户可见提示
   - 优先处理支付、报表、考勤、消息等关键链路

### P2：中优先级

7. 清理历史文档中的“100% 完成”表述
   - 区分 Mock 完成、页面完成、真实联调完成、测试完成
   - 避免答辩或交付时口径不一致

8. 补充测试资产
   - 补充接口联通记录
   - 为关键流程补充自动化或半自动化验证脚本
   - 增加构建后 smoke 检查清单

9. 收口仓库边界
   - 清理历史归档、临时脚本、无关目录
   - 明确哪些内容属于交付物，哪些属于本地辅助产物

---

## 3. 建议执行顺序

1. 真实后端配置与鉴权验证
2. 三端主链路真实联调
3. 修正文档 / OpenAPI / 接口清单差异
4. 收敛服务层与返回模型
5. 增加最小回归测试集
6. 做一次完整构建和演示回归

---

## 4. 风险提示

- 如果继续长期停留在 `DEV_MOCK`，页面“功能完成度”会高估真实可交付程度。
- 当前部分页面依赖 Mock 返回字段的宽松兼容，切到真实后端后最容易暴露字段缺失、状态枚举不一致、分页结构不一致等问题。
- 文档若不及时同步，后续联调、答辩演示和交付说明会出现口径偏差。

---

## 5. 完成标志

满足以下条件后，才建议将项目状态从“Mock 验证完成”升级为“联调完成”：

- `TEST_REAL` 环境可正常登录
- 家长/教师/管理员三端主链路全部走通
- 关键接口返回结构与前端消费字段一致
- 接口文档、OpenAPI、代码常量三者一致
- 完成一次构建验证与一轮回归测试
